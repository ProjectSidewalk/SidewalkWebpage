Sign-in & Sign-up Redesign — Design Comps

Eight token-styled comps for the registration redesign: the native-dialog sign-in and sign-up (default, live validation, and error states), the new /welcome page, mobile parity, the Settings home for username changes and the relocated service-hours opt-in, and the restyled service-hours instructions page. Every color and type style is an existing main.css :root token — nothing new was invented. The markup and CSS here are written to be transplanted directly into navbar.scala.html, welcome.scala.html, and public/css/auth.css.

How to review: the sign-up comp (S3) is live — click into it and type to see the checklist, strength meter, and match indicator respond. The eye toggles work everywhere, and the welcome hero has a replay button for the confetti. Reply with change requests, or a 👍 to unlock the build (WS1+WS2).

Palette — existing tokens only

asphalt-500 ink banana-500 action pine-700 focus pine-600 success error-200 label-crosswalk

Error text is AA-darkened via color-mix(error-200, black) — proposed as a real --color-error-300 token (D4).

Type — system rules kept

Raleway 700 only where the system reserves it: dialog titles (--text-h2-bold), the welcome H1 (--text-h1-bold), and the small-accent eyebrow.

Mulish for everything else — labels, inputs, helpers, cards — via the --text-* tokens at --ui-scale: 1.

Signature — the crosswalk

Every auth surface begins at a crosswalk band — zebra stripes in the Crosswalk label's canonical #FABF1C on asphalt. It's the product's most iconic street feature, drawn in its own label color.

Supporting cast: a sidewalk-slab strength meter (paved as rules pass) and welcome confetti in the seven label-type colors. Everything else stays quiet.

Scope findings from recon

Things the codebase told us that update the #4375 plan (written 2026-06-30, before the dashboard redesign merged):

ItemStatusDetail
WS6 · Username changeshipped Dashboard Settings ships the full rename flow (#4373). Drop WS6 from this epic except the identity-refresh verification.
WS7 · Profanity guardhalf-shipped ProfanityGuard exists (rename + team names) but signUpPost doesn't call it — a bad name gets in at signup and is only caught if the user renames. Remaining work: one guard call + message key in signup.
Username rules driftdecision D3 Signup: letters+numbers only ([a-zA-Z0-9]+$, unanchored). Rename: ^[A-Za-z0-9_-]{3,30}$. Two different products today. Recommend adopting the rename rule everywhere via one shared validator.
Sign-in password gatefix in WS4 The sign-in field carries the signup strength pattern, which can block legitimate older/weaker passwords from ever submitting. Comps drop it (S1).
Service-hours questiondecided: out of signup Empirics per city (dev dump; registered users, YES-rate → ever opened Time Check): Seattle 1,826 · 39.4% → 7.2%; Teaneck 109 · 81.7% → 24.8%; Amsterdam 99 · 45.5% → 6.1%; Vancouver 12 · 75% → 3 users. Even Amsterdam — where US-style service hours barely apply — gets ~46% YES: classic first-option + “count my work?” bias. Teaneck shows the genuine (school) population exists, so the official opt-in stays — relocated to Settings (S7), which points to the restyled instructions (S8). Signup drops the question; the flag, endpoint, and navbar gating already exist; new signups default to off. Mikey's verification loop (feedback emails + Time Check) is untouched.
